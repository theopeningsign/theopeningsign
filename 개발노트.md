## 2. 핵심 문제 (Root Cause)

**Notion API가 주는 이미지 URL은 약 1시간 뒤 만료되는 S3 서명 URL이다.**

- URL 예시: `https://prod-files-secure.s3.us-west-2.amazonaws.com/...?X-Amz-Expires=3600&X-Amz-Signature=...`
- URL 자체에 만료 시각과 서명이 박혀 있어서, 1시간이 지나면 그 주소는 죽는다(403).

### 이게 왜 화면에서 문제가 되나
1. 서버가 Notion 데이터를 `revalidate: 60`(60초)으로 캐싱 → 생성된 HTML 안에 **만료될 S3 URL이 그대로** 박힌다.
2. 이미지 컴포넌트에 `unoptimized={isNotionImageUrl(...)}` 설정 → **Next.js 이미지 최적화를 끔** → 브라우저가 S3 URL을 **직접** 호출한다.
   - (`unoptimized`를 켠 이유: 과거 "Vercel 이미지 최적화 Cache Writes 초과"를 피하려고)
3. 캐시된 페이지를 1시간 넘겨서 열면 그 안의 URL은 이미 죽어 있다 → 이미지 `onError` 발생 → 새로고침/재시도 지옥 시작.

### 증상 (사용자 입장)
- 스피너가 계속 돌고 화면이 "움찔움찔" 거린다.
- 뒤로가기 하면 화면이 통째로 리로드된다.
- 가끔 무한 스피너.
- 전반적으로 프로페셔널하지 못한 느낌.

## 5. 진행할 수정 방향 — "방법 A: 이미지 프록시"

### 핵심 아이디어
**만료되는 S3 URL을 사용자(브라우저)에게 절대 노출하지 않는다.**
우리 서버에 이미지 프록시 라우트를 하나 만들어서, 사용자에겐 안 죽는 우리 주소만 준다.

### 동작 방식 (Before → After)

**[현재]**
```
브라우저 ──(만료될 S3 URL 직접 호출)──> Notion S3
                                          └ 1시간 후 죽음 → onError → 재시도 지옥
```

**[수정 후 — 리다이렉트 방식 확정]**
```
브라우저 ──(안 죽는 우리 주소: /api/img/<pageId>/cover/0)──> 우리 서버(프록시)
                                                              └ Notion에서 싱싱한 S3 URL 받아
                                                                "여기로 가" 하고 307 리다이렉트(튕김)
브라우저 ──(튕긴 S3 주소로 직접)──> Notion S3 에서 사진 바이트 직접 다운로드
```

- 사용자 브라우저엔 `/api/img/...` 같은 **만료 안 되는 내부 주소**만 남는다.
- 프록시 서버는 **바이트를 직접 나르지 않고 리다이렉트만** 한다 → 실제 사진 데이터는 지금처럼 S3→브라우저 직접 전송.
- 프록시 응답(리다이렉트)을 약 50분간 캐싱(S3 만료 1시간 안쪽) → 같은 사진 반복 조회 시 함수 거의 안 깨움.

### Vercel 무료(Hobby) 비용 영향 — 초과 위험 없음 (사용자 우려 반영)
- **이미지 최적화 할당량**: 0 유지. 지금도 `unoptimized`로 우회 중이고 프록시도 최적화 안 씀. (과거 "Cache Writes 초과" 항목 = 이거. 영영 벗어남)
- **대역폭(월 100GB)**: 리다이렉트 방식이라 사진 바이트가 Vercel을 안 거치고 S3→브라우저 직접 → **현재와 거의 동일(≈0 증가)**.
  - ⚠️ 만약 "바이트 스트리밍" 방식으로 했다면 모든 사진이 Vercel을 통과해 대역폭을 잡아먹었을 것. 그래서 리다이렉트로 확정함.
- **함수 호출 수**: "Notion 주소 1회 조회 + 튕기기"만 하는 가벼운 함수 + 50분 캐싱 → 소량. 무료 한도 내 넉넉.

### 그 결과 무엇이 바뀌나
- 이미지가 **언제 봐도** 안정적으로 뜬다. (지금은 "1시간 안에 본 사람만 운 좋게 보임")
- 재시도/강제 리로드/모달이 **필요 없어진다** → 땜빵 7겹 제거 가능.
- 스피너 깜빡임, 화면 움찔거림, 뒤로가기 리로드, 무한 스피너 → 전부 사라짐.
- 코드가 약 1,000줄 줄어 단순해진다.

### 안전장치 (회귀 방지 — "잘 되던 게 깨지는 것" 방지)
1. **순서**: 원인을 먼저 없애고(프록시) → 잘 되는 걸 확인한 뒤에야 땜빵을 제거한다. 제거가 먼저가 아니다.
2. **로딩 표시 정책 (사용자 선호 확정)**:
   - 사진 로드되는 동안 **스피너는 그대로 유지**한다. (사용자가 원함)
   - **placeholder(회색 박스)는 띄우지 않는다.** (사용자가 싫어함)
   - 화면 움찔거림(그리드 전체 숨김→팝인)과 무한 스피너를 유발하던 땜빵은 제거하되, "로딩 중 스피너" 자체는 정상 UX로 남긴다.
   - 만에 하나 이미지가 안 오면: 페이지 전체 리로드/모달 대신, **해당 이미지 1장만 조용히 제한적 재시도**(stable 프록시 URL이라 페이지 흔들림 없음)하며 그동안 스피너 유지. 프록시가 서버에서 싱싱한 URL을 확보하므로 실패 자체가 매우 드묾.
3. **새 브랜치 작업**: main 보존. 미리보기로 검증 후에만 병합. 문제 시 즉시 롤백.
4. **로컬 검증**: 배포 전 직접 띄워서 목록/상세/뒤로가기/라이트박스 이미지 로드를 눈으로 확인.

### 2026-06-09 — ★중요★ ORB 발견 → 프록시 "리다이렉트 → 직접 전달(스트리밍)"로 전환
- **증상**: 데스크톱 크롬에서 일부 포트폴리오 이미지가 무한 스피너. 네트워크 탭에 `net::ERR_BLOCKED_BY_ORB`.
- **원인**: 프록시가 307 리다이렉트로 브라우저를 외부(S3)로 튕겼는데, 크롬 ORB(Opaque Response Blocking)가 "동일출처에서 시작 → 외부로 튕긴 no-cors 응답"을 이미지로 인정 못 하고 차단. (서버측 curl은 200 정상 → 브라우저만 막는 것)
- **헛다리 1 (실패)**: 이미지에 `crossOrigin="anonymous"` 추가해 CORS 모드로 ORB 우회 시도. S3가 `Access-Control-Allow-Origin: *`를 주긴 하나, **리다이렉트를 CORS로 따라가는 과정에서 오히려 전부 CORS error + 403**으로 악화. → 즉시 롤백(crossOrigin 4개 컴포넌트에서 제거).
- **최종 해결 (성공)**: 프록시를 **307 리다이렉트 → 바이트 직접 전달(스트리밍)** 으로 변경. 서버가 S3에서 받아 그대로 내려줌. 브라우저는 동일출처(우리 도메인)하고만 통신 → **ORB/CORS 자체가 발생 불가**. 서버측 검증: `/api/img/...`가 리다이렉트 없이 `200 image/jpeg` 직접 반환 확인.
  - 파일: `src/app/api/img/[...slug]/route.ts` (fetch S3 → `new NextResponse(upstream.body)` 스트리밍, Cache-Control 장기).
- **제약 재확인 (사용자 지적)**: Next 이미지 최적화는 **무료 한도(Image Optimization 5K) 때문에 일부러 끈 것** → 다시 켜면 안 됨. 포트폴리오 이미지는 계속 `unoptimized` 유지. (메인 히어로 `/main/*` 로컬 6장만 최적화 — 영향 미미)
- **새 트레이드오프 = 대역폭**: 이제 이미지가 Vercel을 통과 → Fast Data Transfer(100GB/월) 사용. 예전(S3 직통)은 0이었음. 현재 사용량 802MB/100GB로 여유 큼. 대략 월 6천~1만 방문까지 안전 추정.
  - **레버(아직 미적용)**: 트래픽 커지면 프록시 안에서 `sharp`로 서버측 리사이즈 → 대역폭 ~10배 절감 가능. 이건 Vercel "Image Optimization" 할당량과 무관(우리 함수 CPU만 약간 씀). 지금은 불필요해서 안 함.
- **아이폰 bfcache 문제**: 예전 빡빡한 재시도 로직의 근본 동기 = "아이폰에서 보다가 내려놨다 나중에 보면 S3 URL 만료로 이미지 깨짐". 프록시 URL은 안 죽으므로 **이 문제의 근본 원인이 제거됨**. 단 iOS WebKit bfcache는 실기기 검증 필요 → **배포 후 실제 아이폰(크롬·사파리)에서 "보다가 잠그고 1~2시간 뒤 재접속" 테스트 필수**.
- **검증/배포 계획**: 로컬 데스크톱 확인 → `feature/image-proxy` 브랜치 커밋·푸시 → **Vercel 미리보기 URL**에서 아이폰 테스트(프로덕션 `main` 무영향) → 이상 없으면 병합.
